# Checklist: Government Application Review

## Purpose

Use this checklist to evaluate whether an application screen is suitable for Indonesian government/public-service use.

---

## 1. Institutional Fit

- [ ] The UI feels official and trustworthy.
- [ ] The interface avoids generic SaaS marketing style.
- [ ] The brand color is controlled and not excessive.
- [ ] The page uses clear government/service terminology.
- [ ] The product identity is visible but not overwhelming.

---

## 2. Public-Service Clarity

- [ ] Users can understand what service is provided.
- [ ] Requirements are clear before submission.
- [ ] Processing status is clear.
- [ ] Next steps are explained.
- [ ] Contact/help path is available when relevant.

---

## 3. Administrative Reliability

- [ ] Data entry is structured.
- [ ] Required fields are clear.
- [ ] Validation is strict enough for official data.
- [ ] Review step exists before official submission.
- [ ] Destructive actions require confirmation.
- [ ] Audit-relevant information is visible where needed.

---

## 4. Role and Permission Awareness

- [ ] Navigation matches user role.
- [ ] Restricted actions are hidden or disabled with reason.
- [ ] Approval actions are only visible to authorized roles.
- [ ] User/tenant context is visible.
- [ ] Multi-tenant boundaries are clear.

---

## 5. Document and Verification Workflow

- [ ] Uploaded files show type/size requirements.
- [ ] File upload errors are specific.
- [ ] Verification status is visible.
- [ ] Rejection includes notes/reason.
- [ ] Approved documents are clearly marked.
- [ ] QR/check mechanism is explained if used.

---

## 6. Dashboard and Reporting

- [ ] Metrics are prioritized.
- [ ] Data period is visible.
- [ ] Charts are readable.
- [ ] Tables support search/filter/pagination.
- [ ] Export actions are clear.
- [ ] No fake or decorative metrics are shown.

---

## 7. Copywriting

- [ ] Indonesian copy is clear and official.
- [ ] Technical jargon is minimized.
- [ ] Error messages are actionable.
- [ ] Status terms are consistent.
- [ ] Empty states are helpful.

---

## 8. Accessibility and Inclusion

- [ ] Keyboard navigation works.
- [ ] Text is readable.
- [ ] Color contrast is acceptable.
- [ ] Labels are visible.
- [ ] Status does not rely only on color.
- [ ] Mobile users can complete critical tasks.

---

## 9. Production Readiness

- [ ] Loading/empty/error states exist.
- [ ] Responsive behavior is defined.
- [ ] Components are reusable.
- [ ] Design tokens are used.
- [ ] No arbitrary styling values dominate.
- [ ] Implementation notes are clear.
